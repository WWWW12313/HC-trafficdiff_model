## 1. 事故语义专属扩散模型改造提示词

### 1.1 总体目标

请在现有 CausalDiffTab / TabDiff / TabDDPM 实验基础上，设计并实现一版面向交通事故数据的专属结构化扩散模型。

本次改造不再只按照“连续变量 / 分类变量”做粗粒度扩散，而是围绕交通事故生成任务中最核心的五类变量，分别设计更合适的生成方式、约束方式和评估方式。

核心生成对象包括以下五大类：

1. 事故时间
   - SEASON
   - IS_WEEKEND
   - IS_AM_PEAK
   - IS_PM_PEAK

2. 经纬度空间位置
   - LATITUDE
   - LONGITUDE

3. 车辆类型
   - is_sedan
   - is_suv
   - is_taxi
   - is_truck
   - is_pickup
   - is_bus
   - is_van
   - is_motorcycle
   - is_bicycle
   - TOTAL_VEHICLES
   - IS_MULTI_VEHICLE

4. 事故原因 / 事故类型
   - is_rear_end
   - is_lane_change_related
   - is_pedestrian_involved
   - is_cyclist_involved
   - 以及已有的稳定事故原因指示列

5. 伤亡类别与人数
   - NUMBER_OF_PEDESTRIANS_INJURED_BIN
   - NUMBER_OF_PEDESTRIANS_KILLED_BIN
   - NUMBER_OF_CYCLIST_INJURED_BIN
   - NUMBER_OF_CYCLIST_KILLED_BIN
   - NUMBER_OF_MOTORIST_INJURED_BIN
   - NUMBER_OF_MOTORIST_KILLED_BIN
   - NUMBER OF PERSONS INJURED
   - 如存在死亡人数，也应纳入严重程度建模

路网信息和天气信息不是主要事故结果变量，但不能简单排除在模型之外。它们应根据任务场景采用不同处理方式：

1. 历史迁移评估场景（如 2024 域内、2025 迁移）：优先由生成后的事故时间和经纬度通过真实 OSM / 天气数据进行确定性补全或近邻匹配，并作为校准基准。
2. 未来场景生成（如 2028 / 2030）：没有真实未来天气，路网也可能变化，因此天气和部分路网属性必须支持条件生成或用户场景设定。
3. 训练与消融场景：真实 lookup 结果可作为 correction / calibration 信号，用于约束上下文生成模块。

最终目标是构建一个 accident-aware、road-constrained、causal-regularized、future-simulatable 的结构化事故生成模型。

---

### 1.2 核心建模原则

#### 从 mixed-type diffusion 升级为 accident-aware structured diffusion

当前模型只区分：

- 连续变量：统一走高斯 / EDM 扩散
- 分类变量：统一走离散 / absorbing 扩散

这对通用表格数据是合理的，但对交通事故数据仍然过粗。

请将模型改造成按事故语义分组的多头结构：

1. TimeDiff Head：事故时间生成头
2. GeoRoadDiff Head：道路约束空间生成头
3. VehicleMultiLabel Head：车辆类型多标签生成头
4. CauseMultiLabel Head：事故原因多标签生成头
5. InjuryCountSeverity Head：伤亡类别与人数生成头
6. ContextGenerator / ContextLookup Module：路网与天气上下文模块

不同 head 使用不同的扩散核、loss、约束和后处理逻辑。

五大核心事故变量是主生成对象；天气和路网是上下文变量，可在不同场景中查表、条件生成或校准。

---

### 1.3 事故时间生成逻辑

事故时间只保留四类核心变量：

- SEASON
- IS_WEEKEND
- IS_AM_PEAK
- IS_PM_PEAK

要求：

1. SEASON 使用 4 类离散扩散，但要考虑季节的周期性。
2. IS_WEEKEND 使用 Bernoulli / binary diffusion。
3. IS_AM_PEAK 和 IS_PM_PEAK 使用 binary diffusion，并加入互斥或弱互斥约束。
4. 不再把过细的 TIME_PERIOD 作为核心生成目标。
5. 如后续需要 CRASH_TIME_SIN / CRASH_TIME_COS，可由高峰类别和采样时间段再派生，而不是作为主要扩散目标强行生成。
6. 时间变量应作为天气生成 / 天气查表和事故类型生成的重要条件。

---

### 1.4 经纬度空间生成逻辑

经纬度是本模型最需要重点改造的部分。

目标不是生成任意二维点，而是生成“合理事故发生位置”。

#### 1.4.1 空间生成要求

生成的 LATITUDE / LONGITUDE 必须满足：

1. 位于 NYC 合法边界内。
2. 尽量落在真实道路网络附近。
3. 可以落在路口上。
4. 可以落在两个相邻路口之间的道路中心线附近。
5. 不应生成到湖泊、河流、公园深处、建筑内部或明显无道路区域。
6. 不应只依赖经纬度范围裁剪，因为范围内仍可能存在大量不合理区域。

#### 1.4.2 推荐实现方式

请基于 OSM 路网构建空间候选集：

1. 提取 NYC OSM 道路中心线。
2. 提取所有道路交叉口节点。
3. 对每条相邻路口之间的道路 segment，生成候选点：
   - 路口点
   - 两个路口之间的中心点
   - 必要时加入若干沿道路中心线插值点
4. 训练时将真实事故点投影 / snap 到最近道路中心线或最近候选点。
5. 生成时优先生成候选点索引或道路 segment + segment 内位置，而不是直接生成自由经纬度。
6. 最终输出再还原为 LATITUDE / LONGITUDE。
7. 可允许小范围道路横向扰动，但扰动距离应有限，例如 0-20 米。
8. 对生成点执行空间合法性校验：
   - 到最近道路距离
   - 是否位于水体
   - 是否位于 NYC 边界内
   - 是否偏离 OSM 道路网络过远

#### 1.4.3 推荐扩散方式

优先考虑以下方案之一：

1. Road-candidate categorical diffusion
   - 将道路候选点或 segment ID 作为离散生成对象。

2. Segment-conditioned continuous diffusion
   - 先生成道路 segment，再生成 segment 内归一化位置。

3. Mixture spatial diffusion
   - 用道路候选点作为 mixture center，再生成小范围连续偏移。

不要简单地对 LATITUDE 和 LONGITUDE 分别做独立高斯扩散。

---

### 1.5 车辆类型生成逻辑

车辆类型不是单选类别，而是多标签组合。

要求：

1. 使用 multi-label Bernoulli diffusion 或 binary absorbing diffusion。
2. 每个车辆类型指示列作为一个 Bernoulli 变量。
3. 加入组合一致性约束：
   - 至少存在一种车辆类型
   - TOTAL_VEHICLES 与车辆类型数量一致或近似一致
   - IS_MULTI_VEHICLE 与 TOTAL_VEHICLES 一致
4. 对极稀有车辆类型可合并或降权。
5. 车辆类型应受时间、空间、路网类型影响。
6. 车辆类型应进一步影响事故原因和伤亡结果。

---

### 1.6 事故原因生成逻辑

事故原因 / 事故类型同样是多标签稀疏变量。

要求：

1. 使用 sparse multi-label diffusion。
2. 对事故原因保留宏观稳定类别，不追求过细长尾标签。
3. 使用原因共现图或层级原因树辅助约束。
4. 事故原因应受以下因素影响：
   - 事故时间
   - 经纬度 / 路网类型
   - 天气
   - 车辆类型
5. 事故原因应进一步影响伤亡类别和人数。
6. 对不合理组合加入软约束，例如某些原因与某些车辆类型或道路场景弱相关。

---

### 1.7 伤亡类别与人数生成逻辑

伤亡结果不应简单当成普通连续变量。

请拆成两层。

#### 1.7.1 伤亡类别层

包括：

- 是否有行人受伤 / 死亡
- 是否有骑行者受伤 / 死亡
- 是否有机动车乘员受伤 / 死亡

这些变量使用 binary / ordinal diffusion。

#### 1.7.2 伤亡人数层

NUMBER OF PERSONS INJURED 是大量 0、小整数、长尾分布，不适合普通高斯回归扩散。

优先使用：

1. zero-inflated count diffusion
2. ordinal count diffusion
3. Poisson / Negative Binomial head
4. 分类分桶后再映射回人数

要求：

1. 人数不能为负。
2. 总受伤人数应与各类受伤 bin 一致。
3. 如死亡 bin 为 1，应触发严重程度约束。
4. 伤亡结果应由事故原因、车辆类型、时间、路网和天气共同约束。
5. 伤亡人数是评估核心目标之一，必须单独报告 TSTR、分布距离和 count calibration。

---

### 1.8 路网与天气处理方式：历史查表 + 未来生成双模式

路网和天气不应简单地被视为普通事故结果变量，但也不能完全固定为真实查表变量。

请根据任务场景区分三种模式：

1. historical_lookup 模式
   - 用于 2024 域内测试、2025 迁移测试等已有真实年份数据的场景。
   - 此时路网和天气优先由真实 OSM / 天气数据按时间和经纬度补全。

2. future_simulation 模式
   - 用于 2028、2030 等未来事故场景生成。
   - 此时没有真实未来天气，路网也可能变化，因此天气和部分路网上下文需要由条件模型生成或由用户设定场景。

3. correction / calibration 模式
   - 用于训练和评估时做对比修正。
   - 模型先生成或预测路网/天气上下文，再与真实 lookup 结果对比，用误差作为校准信号，而不是把 lookup 当成唯一来源。

#### 1.8.1 路网信息

路网变量包括：

- DIST_TO_SIGNAL_M
- HAS_TRAFFIC_SIGNAL
- INFERRED_LANES
- OSM_TYPE
- OSM_ONEWAY
- HAS_DIVIDER

路网的处理应分两层。

第一层：道路几何约束。

无论是历史年份还是未来年份，经纬度生成都必须受道路网络约束。

要求：

1. 生成点应绑定到 road segment、路口节点或道路中心线候选点。
2. 不允许自由生成到湖泊、河流、公园深处、建筑内部等明显非道路区域。
3. 如果有目标年份对应的 OSM / 规划路网，则使用目标年份路网。
4. 如果没有目标年份路网，则使用最近年份 OSM 作为基础路网，并允许通过 scenario 参数模拟新增道路、封闭道路、车道变化等情况。

第二层：路网属性上下文。

路网属性不建议作为完全自由扩散变量。

优先级如下：

1. 若 road segment 已知，则直接从 segment 继承 OSM_TYPE、OSM_ONEWAY、INFERRED_LANES 等属性。
2. 若做未来模拟且路网可能变化，则允许使用 scenario-conditioned road context generator。
3. 若没有可用路网属性，则使用条件生成模型估计路网上下文，但必须通过道路合法性校验。
4. 历史评估时，模型生成的路网上下文应与真实 lookup 结果对比，用于 calibration / correction。

因此，路网不是“完全不扩散”，而是：

- 道路几何：强约束 / 候选集生成
- 路网属性：优先由 road segment 派生，未来场景下可条件生成或场景设定
- 真实年份：可作为修正和评估基准

#### 1.8.2 天气信息

天气变量包括：

- TEMP_C
- prcp
- WIND_SPEED_KMH
- WEATHER_CONDITION

天气不能只依赖真实查表，因为未来年份没有真实观测值。

请支持两种天气来源。

第一种：历史 lookup 天气。

用于已有年份：

1. 2024 域内测试
2. 2025 迁移测试
3. 其他已有观测年份

处理方式：

1. 根据生成的事故时间和经纬度匹配真实天气数据。
2. 真实天气作为评估基准。
3. 可用于校准 weather generator。

第二种：条件天气生成。

用于未来年份或场景模拟：

1. 根据 SEASON、IS_WEEKEND、IS_AM_PEAK、IS_PM_PEAK、经纬度、月份或气候区生成天气。
2. 天气生成应使用 conditional weather generator，而不是无条件随机采样。
3. WEATHER_CONDITION 可用 categorical / ordinal diffusion。
4. TEMP_C、prcp、WIND_SPEED_KMH 可用条件连续扩散或分布采样。
5. 需保持物理一致性：
   - Snow 通常对应低温
   - Rain 对应 prcp > 0 的概率更高
   - Clear 对应 prcp 接近 0
   - 极端风速和降水应符合季节分布
6. 未来场景允许用户指定天气条件，例如：
   - normal climate
   - heavy rain scenario
   - snow scenario
   - heat wave scenario
   - extreme weather scenario

因此，天气不是事故核心生成目标，但在未来模拟中必须可生成。它属于“条件上下文生成变量”。

### 1.9 因果结构设计

请围绕以下主链条构建宏观软因果骨架：

1. 事故时间 -> 天气
2. 经纬度 / road segment -> 路网属性
3. 事故时间 -> 车辆类型
4. 经纬度 / 路网 -> 车辆类型
5. 天气 / 路网 -> 事故原因
6. 车辆类型 -> 事故原因
7. 事故时间 -> 事故原因
8. 事故原因 -> 伤亡类别
9. 车辆类型 -> 伤亡类别
10. 事故原因 -> 伤亡人数
11. 伤亡类别 -> 伤亡人数

注意：

1. 天气和路网不是五大核心事故结果变量，但它们是重要上下文变量。
2. 在 historical_lookup 模式下，天气/路网主要由真实数据补全。
3. 在 future_simulation 模式下，天气/部分路网上下文需要条件生成。
4. 在 correction 模式下，生成上下文与真实 lookup 上下文进行对比校准。
5. 因果约束应使用 soft mask，不要使用过强硬约束。
6. 因果约束强度建议从 lambda=0.1 / 0.3 / 0.5 做扫描。
7. 对跨年份不稳定的边应弱化。
8. 对长尾事故原因边应降权。

### 1.10 推荐模型结构

实现一个新的模型配置，例如：

- model_name: traffic_structured_diffusion
- experiment_id: traffic_structured_2024
- generation_targets:
  - time
  - road_constrained_location
  - vehicle_type
  - crash_cause
  - injury_outcome
- context_variables:
  - road_features
  - weather_features
- context_mode:
  - historical_lookup
  - future_simulation
  - correction
- causal_context:
  - road_features
  - weather_features

模型流程根据模式区分。

#### 1.10.1 historical_lookup 模式

用于 2024/2025 这类已有真实数据年份：

1. 生成事故时间四类变量。
2. 生成道路约束经纬度。
3. 由时间 + 经纬度查真实天气。
4. 由经纬度 / road segment 查真实路网。
5. 基于时间 + 经纬度 + 路网 + 天气生成车辆类型。
6. 基于时间 + 经纬度 + 路网 + 天气 + 车辆类型生成事故原因。
7. 基于事故原因 + 车辆类型 + 时间 + 路网 + 天气生成伤亡类别。
8. 基于伤亡类别 + 事故原因 + 车辆类型生成伤亡人数。
9. 输出完整融合表。

#### 1.10.2 future_simulation 模式

用于 2028、2030 等未来场景：

1. 生成或指定事故时间。
2. 使用目标年份路网、最近年份路网或用户设定路网场景。
3. 生成道路约束经纬度。
4. 根据时间 + 经纬度 + scenario 生成天气上下文。
5. 根据 road segment 派生路网属性，缺失部分由条件模型补全。
6. 基于时间 + 空间 + 生成天气 + 路网生成车辆类型。
7. 基于车辆类型 + 天气 + 路网生成事故原因。
8. 基于事故原因 + 车辆类型 + 天气 + 路网生成伤亡类别和人数。
9. 输出未来事故场景合成数据。

#### 1.10.3 correction 模式

用于训练和消融：

1. 模型先生成天气/路网上下文。
2. 同时用真实 lookup 得到天气/路网。
3. 计算生成上下文与真实上下文之间的 calibration loss。
4. 评估“直接查表”和“条件生成”对事故原因、车辆类型、伤亡结果的影响。
5. 用于验证未来模拟时上下文生成模块是否可靠。

### 1.11 评估要求

评估必须分别报告五大核心生成对象的质量。

#### 1.11.1 时间评估

- SEASON 分布 JS
- IS_WEEKEND 分布差异
- IS_AM_PEAK / IS_PM_PEAK 分布差异
- 时间组合合法性

#### 1.11.2 空间评估

- 经纬度 Wasserstein / MMD
- 到最近道路距离分布
- 生成点落在道路附近比例
- 生成点落在水体 / 非道路区域比例
- 路口点与路段中点覆盖情况
- borough / OSM_TYPE 分布一致性

#### 1.11.3 车辆类型评估

- 每类车辆边际分布
- 多标签组合分布
- TOTAL_VEHICLES 一致性
- IS_MULTI_VEHICLE 一致性

#### 1.11.4 事故原因评估

- 原因标签边际分布
- 原因共现矩阵
- 原因-车辆类型联合分布
- 原因-路网联合分布

#### 1.11.5 伤亡评估

- 伤亡 bin 分布
- 伤亡人数分布
- zero ratio
- count MAE / calibration
- TSTR 对 NUMBER OF PERSONS INJURED 的预测性能

#### 1.11.6 路网 / 天气上下文评估

- historical_lookup 模式下：报告 lookup 成功率、缺失率、fallback 比例。
- future_simulation 模式下：报告天气物理一致性、季节一致性、极端值比例。
- correction 模式下：报告生成上下文与真实 lookup 上下文的误差。

#### 1.11.7 迁移评估

必须同时报告：

1. 2024 域内测试
2. 2025 跨年份迁移测试
3. 2024 -> 2025 性能退化率
4. 五大核心变量在 2025 上的分布一致性
5. 与以下 baseline 对比：
   - baseline_tabddpm
   - macro_soft_2024
   - ablation_no_causal
   - ablation_no_hierarchy
   - SMOTE
   - CTGAN
   - TVAE

### 1.12 消融实验设计

至少包含以下消融：

1. 去掉道路约束空间生成，只用普通经纬度高斯扩散。
2. 路网/天气只做 historical lookup，不做 future context generation。
3. 路网/天气完全作为普通扩散目标，不使用真实 lookup 或 road segment 派生。
4. 使用 future_simulation weather generator，但不做 correction calibration。
5. 去掉因果 soft mask。
6. 去掉多头结构，退回普通 num/cat mixed diffusion。
7. 去掉伤亡 count 专用 head，退回普通回归/连续扩散。
8. 去掉车辆/原因 multi-label 约束。

重点验证：

1. 道路约束是否减少不合理空间点。
2. 历史 lookup 是否提升 2024/2025 评估质量。
3. 条件天气生成是否使模型具备未来年份模拟能力。
4. correction calibration 是否提升 future_simulation 的可靠性。
5. 专属 head 是否提升五大核心变量的生成质量。
6. 因果约束是否降低 2024->2025 迁移退化。
7. 伤亡人数专用建模是否提升 TSTR 和 count 分布拟合。

### 1.13 预期结论方向

本模型应证明：

1. 对交通事故数据，单纯 mixed-type tabular diffusion 不够。
2. 经纬度必须使用道路网络约束，否则会生成湖泊、河流、非道路区域等不合理事故点。
3. 路网和天气不能简单处理为普通随机表格变量。
4. 对已有历史年份，路网和天气适合用真实 lookup 作为上下文与校准基准。
5. 对未来年份，天气和部分路网属性必须支持条件生成或场景设定，否则模型没有未来模拟价值。
6. 车辆类型、事故原因、伤亡类别本质是多标签结构，适合 binary / multi-label diffusion。
7. 伤亡人数是零膨胀小整数计数变量，适合 count-aware diffusion 或专用 count head。
8. 事故时间、空间、车辆、原因、伤亡五大对象的专属生成机制，比通用 num/cat 两类扩散更符合交通事故机理。
9. 宏观软因果骨架应作为跨组约束，而不是替代各变量自身的最佳扩散机制。

最终目标是构建一个 accident-aware、road-constrained、causal-regularized、future-simulatable 的结构化事故生成模型。

## 2. 执行优先级建议

### 2.1 第一阶段：最小可行版本

1. 保留当前 TabDiff 主干。
2. 先不重写完整扩散核，只新增道路约束经纬度后处理 / 候选点吸附模块。
3. 将天气和路网从“主扩散目标”中移出，改成可配置的 context_mode。
4. 增加五大核心变量的专项评估指标。

### 2.2 第二阶段：多头结构

1. 拆分 TimeDiff、GeoRoadDiff、Vehicle、Cause、Injury 五个输出头。
2. 对车辆和原因使用 multi-label loss。
3. 对伤亡人数增加 zero-inflated / count-aware head。
4. 增加上下文校准 loss。

### 2.3 第三阶段：未来模拟能力

1. 实现 future_simulation context generator。
2. 支持 normal / rain / snow / heat wave / extreme weather 等 scenario。
3. 支持最近年份 OSM + 用户设定路网变化。
4. 与 historical_lookup 和 correction 模式做消融对比。
