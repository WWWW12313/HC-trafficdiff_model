### 正式评价指标

最终统一评估由 `pipeline/run_finish_pipeline.py --tier full` 触发，内部调用 `pipeline/evaluate_all.py`，分两套测试集输出报告：

- 2024 域内：`results/eval_report_full_source2024_in_domain.*`。
- 2025 迁移：`results/eval_report_full_source2024_transfer2025.*`。
- 综合汇总：`results/eval_full_source2024_compare.md`。

核心指标口径：

- `mean_wasserstein_numeric`：连续变量边际分布距离，越低越好。
- `mean_js_categorical`：分类变量 Jensen-Shannon 距离，越低越好。
- `tstr_avg_score`：TSTR 下游任务平均分，用合成训练、真实测试，越高越好。
- `tstr_best_model` / `tstr_r2` / `tstr_mse` / `tstr_mae`：TSTR 最优下游模型及回归质量指标。
- `primary_metrics_profile=no_rule`：正式主指标暂不把规则违背率作为主排序项，聚焦分布保真与迁移 TSTR。

### Stage 1/2/3 结构与域内消融诊断

当前分层结构不是三个都训练扩散模型，而是“两段模型 + 一段确定性上下文”：

| Stage | 角色 | 变量 | 当前实现 |
| --- | --- | --- | --- |
| Stage 1 | 时空锚点 | `LATITUDE/LONGITUDE/CRASH_TIME_SIN/CRASH_TIME_COS` + `SEASON/DAY_OF_WEEK/TIME_PERIOD` | `train_hierarchical.py --stage 1` 训练 `nyc_stage1_2024` |
| Stage 2 | 离线 API/OSM/天气上下文 | `TEMP_C/prcp/WIND_SPEED_KMH/DIST_TO_SIGNAL_M/REAL_SPEED_LIMIT/INFERRED_LANES` + `HAS_TRAFFIC_SIGNAL/OSM_ONEWAY/HAS_DIVIDER/coco/WEATHER_CONDITION/OSM_TYPE` | 不单独训练；由 `prepare_2025_data.py` 基于坐标/时间查离线天气与 OSM |
| Stage 3 | 事故微观结果 | 车辆类型、原因类型、伤亡 bin、`TOTAL_VEHICLES/IS_MULTI_VEHICLE` 和目标 `NUMBER OF PERSONS INJURED` | `train_hierarchical.py --stage 3` 训练全表扩散模型 |

四组实验实际差异：

- `baseline_tabddpm`：无因果、无分层，Stage 3 全表 unconditional 生成。
- `ablation_no_hierarchy`：有因果掩码、无分层，Stage 3 全表 unconditional 生成。
- `ablation_no_causal`：无因果、有分层，采样时使用 `impute_stage3`。
- `ours_full_model`：有因果、有分层，采样时使用 `impute_stage3`。

域内消融结果不理想的主要原因判断：

1. `impute_stage3` 采样会随机选训练集真实行作为条件，冻结 Stage 1+2，只重采样目标 `y` 和 Stage 3 分类列。也就是说，`ours_full_model` 与 `ablation_no_causal` 的时空、天气、OSM 上下文大部分来自真实训练行，域内 TSTR 容易被真实条件行托高，因果掩码的边际收益被掩盖。
2. `ours_full_model` 和 `ablation_no_causal` 的采样条件相同，差异主要只剩 Stage 3 训练时的因果正则；如果 causal mask 与目标任务不完全对齐，或者正则过强，域内拟合可能略受约束。
3. 当前 2024 训练表中 `REAL_SPEED_LIMIT` 和 `HAS_DIVIDER` 都是常数：`REAL_SPEED_LIMIT=-0.128317...`、`HAS_DIVIDER=0`。这会削弱 Stage 2 路网特征的信息量，也会让因果 mask 中涉及 `REAL_SPEED_LIMIT` 的边没有实际贡献。
4. 2024 域内测试与训练分布接近，简单 unconditional 的 `baseline_tabddpm/ablation_no_hierarchy` 已能拟合得很好；完整模型的优势更容易体现在 2025 迁移测试中，而不是域内测试。
5. 采样评估中 `ours_full_model` 的 `tstr_best_model_score` 仍略高，但 `tstr_avg_score` 被 MLP 等较不稳定模型拉低；因此域内排名不宜只看综合平均，需要同时看 best-model 与迁移结果。

已修正 `prepare_2025_data.py` 的 `REAL_SPEED_LIMIT` 来源：现在优先解析最近 OSM 边的 `maxspeed` 标签并转为 mph；缺失时才按道路等级/NYC 默认限速 fallback（如 residential 25、primary/secondary 30、trunk 40、motorway 50）。OSM 文件使用口径保持不变：2024 训练集使用 `raw_data/osm/2024` 下由 `new-york-250101.osm.pbf` 派生的 graphml；2025 迁移测试使用 `raw_data/osm/2025` 下由 `new-york-260101.osm.pbf` 派生的 graphml。

下一步若要判断域内指标是否优化，需要重建 `nyc_crash_2024/nyc_stage1_2024` 与 `nyc_crash_2025`，再重训四组 full 模型并重新评估；旧报告不能代表修复后 speed-limit 版本。

### 下一轮改进实验提示词：Stage2 可训练化 + NOTEARS 因果图修正

#### 背景判断

当前 2024-source 正式实验已经完成，但完整模型的分层采样仍主要采用 `impute_stage3`：随机选取真实训练行作为条件，冻结 Stage1+Stage2，只重采样目标 `y` 和 Stage3 分类列。这一设置适合“给定真实测试事故上下文的迁移评估”，但不完全适合“未来年份事故数据生成/预测”。

如果目标是生成未来事故数据，则未来没有真实事故行可供参照，`时空、天气、OSM 上下文` 不能长期依赖真实训练行或测试行。Stage2 需要从“离线查表/条件上下文层”升级为“可训练的条件生成层”。

#### 总目标

请基于现有 CausalDiffTab/TabDiff 代码线，设计并实现下一轮改进实验：

```text
Stage1: P(time, space)
Stage2: P(weather, road | time, space)
Stage3: P(vehicle, cause, injury | time, space, weather, road)
```

即：

1. Stage1 生成事故时空锚点。
2. Stage2 在 Stage1 条件下生成天气与 OSM/路网上下文。
3. Stage3 在 Stage1+Stage2 条件下生成车辆类型、事故原因、伤亡类别和受伤人数目标。

最终目标是让模型具备“未来无真实事故行条件”下的完整链式生成能力，而不是只做 Stage3 条件补全。

#### Stage 划分建议

##### Stage1：时空锚点层

保留：`LATITUDE`、`LONGITUDE`、`CRASH_TIME_SIN`、`CRASH_TIME_COS`、`SEASON`、`DAY_OF_WEEK`、`TIME_PERIOD`。

说明：Stage1 是未来生成的初始锚点；可继续训练 `nyc_stage1_2024`，但后续采样不应只从真实训练行取条件，而应支持 Stage1 unconditional 生成。

##### Stage2：环境/路网上下文生成层

新增可训练 Stage2 数据集与模型。

连续变量：`TEMP_C`、`prcp`、`WIND_SPEED_KMH`、`DIST_TO_SIGNAL_M`、`REAL_SPEED_LIMIT`、`INFERRED_LANES`。

分类变量：`HAS_TRAFFIC_SIGNAL`、`OSM_ONEWAY`、`HAS_DIVIDER`、`coco`、`WEATHER_CONDITION`、`OSM_TYPE`。

训练形式：

```text
Stage2 target = weather + road context
Stage2 condition = Stage1 time + space
```

实现要求：

1. 新增 `nyc_stage2_2024` 数据目录。
2. 新增 Stage2 causal masks。
3. `train_hierarchical.py` 支持 `--stage 2`。
4. `sample_conditional.py` 支持 Stage1→Stage2 条件采样。
5. 对未来生成任务，Stage2 不再直接引用真实训练行的天气/OSM 上下文。

##### Stage3：事故微观结果层

Stage3 继续生成车辆类型二值列、事故原因二值列、伤亡 bin、`TOTAL_VEHICLES`、`IS_MULTI_VEHICLE` 和目标 `NUMBER OF PERSONS INJURED`。

采样形式需要从当前：

```text
真实训练行 Stage1+2 -> impute Stage3
```

改进为：

```text
生成 Stage1 -> 生成 Stage2 -> 条件生成 Stage3
```

#### 因果图修正规则

NOTEARS 算法发现的图只能作为候选统计结构，不能直接作为最终因果图。最终图必须经过领域规则修正。特别注意：事故时间影响路网属性的边不合理，应强制删除。

##### 强制删除边

- `CRASH_TIME_SIN/CRASH_TIME_COS/SEASON/DAY_OF_WEEK/TIME_PERIOD -> OSM 路网属性`
- `天气变量 -> 静态 OSM 路网属性`
- `伤亡结果 -> 时间/空间/天气/路网`
- `事故原因/伤亡结果 -> 车辆类型`
- `事故结果变量 -> Stage1/Stage2 上下文变量`

其中 OSM 路网属性包括：`DIST_TO_SIGNAL_M`、`REAL_SPEED_LIMIT`、`INFERRED_LANES`、`HAS_TRAFFIC_SIGNAL`、`OSM_ONEWAY`、`HAS_DIVIDER`、`OSM_TYPE`。

原因：路网属性来自 OSM 快照和经纬度空间匹配；事故发生时间不会导致限速、车道数、道路类型、单行属性变化；2024 与 2025 使用不同 OSM 快照，这是年份/场景层面的外生设置，不应被建模成单条事故样本中的时间→路网因果边。

##### 强制保留边

- `LATITUDE/LONGITUDE -> OSM 路网属性`
- `时间/季节 -> 天气`
- `天气/路网 -> 事故原因`
- `时间 -> 事故原因`
- `车辆类型 -> 事故原因`
- `车辆类型 -> TOTAL_VEHICLES/IS_MULTI_VEHICLE`
- `事故原因 -> 伤亡结果`
- `车辆类型 -> 伤亡结果`
- `天气/路网 -> 伤亡结果`，建议弱约束

##### 可选场景节点

如果后续显式加入年份或路网快照变量，可使用 `YEAR / OSM_SNAPSHOT -> OSM 路网属性`，但不要用事故时间变量代替年份/快照变量。

#### 宏观因果结构

下一版仍应沿用“宏观骨架 + soft mask”的思路，不回退到旧版 222 边密集图。

推荐宏观结构：

```text
时间/空间锚点 -> 天气/路网环境 -> 事故原因/行为 -> 伤亡结果
车辆类型 -> 事故原因/行为 -> 伤亡结果
空间坐标 -> OSM 路网属性
```

可采用三档权重：强保留边 `1.0`，弱约束边 `0.5`，删除边 `0.0`。
